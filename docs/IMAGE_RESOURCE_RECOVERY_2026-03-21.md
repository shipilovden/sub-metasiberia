# Image Resource Recovery

Дата: 2026-03-21

## Что исправлено

- Путь создания `image object` теперь использует ту же PNG-нормализацию, что и обычная загрузка объекта.
- `GUIClient` для world-объектов снова использует тот же basis/LOD путь, что и upstream `substrata`, без отдельной special-case ветки для `image_cube`.
- Сервер при чтении `server_state.bin` перепроверяет, существует ли backing file ресурса на диске, и сбрасывает stale-ресурсы в `State_NotPresent`.
- При логине сервер повторно запрашивает у владельца только отсутствующие базовые ресурсы, а не слепо доверяет старому состоянию в БД.
- После старта сервера автоматически ставится в очередь регенерация texture derivatives для уже существующих объектов, world settings и avatar materials.
- `EXR` переведён в safe-path без обязательных `.basis/_lod` производных, как `WEBP/BMP/TGA/TIF/TIFF`.

## Зачем это нужно

- Старые миры могли содержать ссылки на ресурсы, которые остались в `server_state.bin`, но были удалены из `server_resources`.
- При включённой basis-capability клиент мог запрашивать отсутствующие `.basis` и показывать белые плоскости.
- Разные client upload paths раньше по-разному обрабатывали `WEBP/BMP/TGA/TIFF`, из-за чего drag-and-drop и диалог давали разный результат.

## Практический эффект

- Новые загрузки изображений через диалог и `image object` идут единообразно.
- Старые миры после деплоя и перезапуска сервера получают шанс на автоматическое восстановление missing originals и missing derivatives без ручной чистки БД.
- `image_cube`-панели снова проходят через тот же клиентский путь выбора текстур, что и в родительском `substrata`.
